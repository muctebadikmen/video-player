# v1.2.0 — Subtitle Sync · Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax. Pure-core tasks are TDD (red → green → commit); Android/UI tasks gate on a successful build + an on-device check.

**Goal:** Ship **feature C** of the subtitle-power design as **Release v1.2.0** — manual subtitle sync for external/downloaded subtitles with three controls: ① **Delay** (±50 ms nudge, already exists), ② **Speed/rate** correction (±0.001 fine / ±0.01 coarse steppers, current value shown), and ③ a guided **Precise sync (two-point)** flow that captures two cue lines and fits a `rate` + `offset` automatically. The chosen rate persists per-file alongside the existing `subtitleOffsetMs`, and live preview runs through the existing `CueOverlay`.

**Architecture:**
- **Pure core (`:core:playback`)** owns the timing math: the active-cue lookup gains a `rate` parameter (`effectiveTime = (positionMs * rate).toLong() + offsetMs`), plus a `twoPointSync(orig1, want1, orig2, want2)` linear fit returning `SyncResult(rate, offset)`, plus `DEFAULT_SUBTITLE_RATE`. No Android, TDD-first.
- **Persistence (`:app`)** adds a per-file `subtitleRate: Float` column to `PlaybackMemoryEntity`, threaded through `ResolvedStartSettings`, `PlaybackMemoryRepository.persistSubtitle`/`resolveStart`, and `PlayerViewModel.persistSubtitle`, exactly mirroring `subtitleOffsetMs`. Because `subtitleRate` is a **brand-new** column (the reserved columns were `audioTrackId`/`subtitleTrackId`/`subtitleOffsetMs`/`orientation`/`v2*` — there is **no** reserved `subtitleRate`), the Room DB version bumps **1 → 2** with a real `Migration(1, 2)` adding `subtitleRate REAL NOT NULL DEFAULT 1.0`. No destructive migration.
- **UI (`:app`)** extends the CC dropdown (gated on `selectedSubtitleUri != null`, i.e. external/downloaded subs only): existing Delay rows stay; new Speed steppers and a Precise-sync (two-point) guided flow are added. New `PlayerScreen` state (`subtitleRate`, two-point capture state) mirrors the offset wiring (restore from `resolved`, persist immediately on change).

**Tech Stack:** Kotlin, Jetpack Compose (Material3), AndroidX Media3 (ExoPlayer), Room + DataStore. Pure timing logic in `:core:playback`; all Android/Room/Compose in `:app`. Truth + JUnit + Robolectric for tests (already on the classpath).

**Design notes (load-bearing):**
- `effectiveTime = (positionMs * rate).toLong() + offsetMs`. The existing 3-arg `activeCueText(cues, positionMs, offsetMs)` becomes the 4-arg form with `rate` defaulted to `DEFAULT_SUBTITLE_RATE`, so all current call sites (the test at `SubtitleTest.kt:31-40` and `PlayerScreen.kt:713`) compile and behave identically when `rate == 1.0`.
- `twoPointSync` fits the line `want = rate*orig + offset` through the two captured points:
  `rate = (want2 - want1).toDouble() / (orig2 - orig1).toDouble()`, then `offset = want1 - (orig1 * rate).toLong()`.
- **Degenerate guard (decision: return identity, do not throw):** when `orig2 == orig1` the slope is undefined; return `SyncResult(DEFAULT_SUBTITLE_RATE, 0L)` (identity — leaves timing unchanged) rather than throwing, because the call site is UI-driven and a crash on a fat-finger double-tap of the same line is a worse UX than a no-op. This is asserted in the tests below.

/** Identity playback rate for subtitle sync: 1.0 = the subtitle file's own timeline, unscaled. */
const val DEFAULT_SUBTITLE_RATE: Double = 1.0

/** A linear subtitle-timing correction: effective time = position*[rate] + [offset] ms. */
data class SyncResult(val rate: Double, val offset: Long)

/**
 * Text of the cue active at [positionMs] after applying a linear timing correction:
 * `effective = (positionMs * rate).toLong() + offsetMs`. [rate] > 1 makes cues arrive
 * later relative to playback (stretches the subtitle timeline); [offsetMs] > 0 makes
 * cues appear earlier (a constant shift). Returns null when no cue covers that time.
 * Linear scan — cue lists are modest and this runs on the position tick; fine for V1.
 */
fun activeCueText(
    cues: List<SubtitleCue>,
    positionMs: Long,
    offsetMs: Long,
    rate: Double = DEFAULT_SUBTITLE_RATE,
): String? {
    val t = (positionMs * rate).toLong() + offsetMs
    return cues.firstOrNull { t >= it.startMs && t < it.endMs }?.text
}

/**
 * Fit a linear correction from two matched points. The user marks two subtitle lines: for each,
 * [orig1]/[orig2] is the real playback time the line should appear at (when the user taps "Mark"),
 * and [want1]/[want2] is that cue's original start time in the subtitle file. We solve
 * `want = rate*orig + offset` so that [activeCueText] reproduces the marks:
 *   rate   = (want2 - want1) / (orig2 - orig1)
 *   offset = want1 - (orig1 * rate).toLong()
 * Degenerate input ([orig1] == [orig2], i.e. both lines marked at the same instant) yields an
 * undefined slope; we return the identity correction (no change) rather than throwing, because
 * the caller is a UI gesture and a no-op is the safer response.
 */
fun twoPointSync(orig1: Long, want1: Long, orig2: Long, want2: Long): SyncResult {
    if (orig2 == orig1) return SyncResult(DEFAULT_SUBTITLE_RATE, 0L)
    val rate = (want2 - want1).toDouble() / (orig2 - orig1).toDouble()
    val offset = want1 - (orig1 * rate).toLong()
    return SyncResult(rate, offset)
}
```

/**
 * The original file-start time (ms) of the cue the user is marking for two-point sync. Prefers the
 * cue currently on screen at [positionMs] under the active correction; if none is showing, falls back
 * to the next upcoming cue's start (so a tap slightly before a line still captures that line). Null
 * when there are no cues at/after the position.
 */
private fun cueStartForMark(
    cues: List<com.videoplayer.core.playback.SubtitleCue>,
    positionMs: Long,
    offsetMs: Long,
    rate: Double,
): Long? {
    val t = (positionMs * rate).toLong() + offsetMs
    cues.firstOrNull { t >= it.startMs && t < it.endMs }?.let { return it.startMs }
    return cues.filter { it.startMs >= t }.minByOrNull { it.startMs }?.startMs
}
```
